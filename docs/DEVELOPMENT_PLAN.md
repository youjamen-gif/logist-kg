# DEVELOPMENT_PLAN.md — Logist.kg

Документ описывает стандартный процесс разработки платформы Logist.kg.

Он создан специально для проекта, который активно разрабатывается с использованием AI.

Этот документ фиксирует:

- порядок разработки функций
- правила внесения изменений
- правила работы с документацией
- правила генерации кода через AI
- правила контроля архитектуры

---

# 1. Главный принцип разработки

Разработка платформы Logist.kg должна происходить по принципу:

```text
Architecture → Documentation → Implementation

Это означает:

сначала определяется архитектура

затем обновляется документация

только потом создаётся код

AI не должен писать код без архитектурного контекста.

2. Источники истины проекта

Перед началом разработки AI обязан проверить документацию.

Главные документы проекта:

docs/AI_RULES.md
docs/PROJECT_STRUCTURE.md
docs/BACKEND_ARCHITECTURE.md
docs/DATABASE_SCHEMA.md
docs/DATABASE_SCHEMA_SQL.md
docs/AUTH_FLOW.md
docs/API_SPEC_BACKEND.md
docs/ADMIN_PANEL_STRUCTURE.md

Если код противоречит документации — документация считается приоритетом.

3. Стандартный цикл разработки функции

Каждая новая функция должна разрабатываться по следующему циклу.

Шаг 1 — определить домен

AI должен определить:

к какому модулю относится функция

какие сущности участвуют

какой API будет использоваться

Пример:

функция: создание ставки
модуль: bids
сущности: bids, freights, users
Шаг 2 — проверить архитектуру

AI должен проверить:

есть ли уже похожая функция

соответствует ли функция API_SPEC

требуется ли изменение БД

требуется ли новый endpoint

Шаг 3 — обновить документацию

Если новая функция требует изменений:

API_SPEC_BACKEND.md

DATABASE_SCHEMA.md

ADMIN_PANEL_STRUCTURE.md

AUTH_FLOW.md

документация должна быть обновлена до написания кода.

Шаг 4 — создать backend endpoint

Backend должен быть создан первым.

Backend включает:

controller
service
dto
guards
serializer

Frontend не должен реализовывать функциональность без backend.

Шаг 5 — создать frontend интеграцию

Frontend реализует:

UI

формы

API вызовы

отображение данных

Frontend не должен содержать бизнес-логику.

Шаг 6 — проверить безопасность

AI обязан проверить:

role checks

ownership checks

DTO validation

audit logging при необходимости

Шаг 7 — проверить производительность

AI обязан проверить:

pagination

индексы

размер ответа

количество запросов

4. Правила разработки backend

Backend должен разрабатываться строго по модульной архитектуре.

Стандарт структуры модуля:

module
├── module.controller.ts
├── module.service.ts
├── module.module.ts
├── dto
├── serializers
5. Правила разработки frontend

Frontend должен соблюдать следующие правила:

страницы остаются тонкими

бизнес-логика в backend

API слой централизован

компоненты переиспользуемы

UI разделён на domain components

6. Правила изменения базы данных

Если требуется изменение БД:

обновляется DATABASE_SCHEMA.md

обновляется DATABASE_SCHEMA_SQL.md

обновляется Prisma schema

создаётся миграция

проверяется API

AI не должен менять базу данных без обновления документации.

7. Правила добавления API

При создании нового endpoint AI обязан:

добавить endpoint в API_SPEC_BACKEND.md

описать request

описать response

описать доступ (roles)

описать ограничения

Только после этого создаётся код.

8. Правила admin-функций

Все admin функции должны:

использовать /admin API

использовать admin guards

логироваться в audit_logs

иметь confirm dialog

9. Правила добавления новых модулей

Новый модуль можно создать только если:

функция не относится к существующему модулю

модуль имеет чёткую доменную границу

AI не должен создавать десятки мелких модулей.

10. Правила рефакторинга

Рефакторинг допускается если:

код дублируется

архитектура нарушена

код трудно поддерживать

Но refactor не должен:

ломать API

ломать структуру БД

ломать frontend

11. Правила работы с AI

AI должен:

проверять документацию

писать код по архитектуре

не упрощать безопасность

не создавать хаос в структуре

не копировать код без анализа

AI не должен:

генерировать случайный код

менять архитектуру без обновления docs

писать временные решения

12. Правила коммитов

Каждый коммит должен содержать понятное описание.

Примеры:

feat: add freight creation endpoint
fix: correct ownership guard in bids module
refactor: split admin services
docs: update API spec for orders
13. Проверка перед merge

Перед интеграцией изменений необходимо проверить:

архитектура
безопасность
пагинация
DTO validation
role guards
ownership guards
документация
14. Разработка через AI

Если код генерируется AI:

AI должен:

проверить архитектуру

проверить документацию

написать код

проверить безопасность

проверить производительность

15. Масштабируемость

Каждая новая функция должна учитывать будущий рост платформы.

Система должна быть готова к:

100k пользователей
100k грузов
1000+ realtime соединений
16. Главный принцип

Разработка должна быть:

предсказуемой
архитектурной
безопасной
масштабируемой

Если изменение:

ломает архитектуру

обходится без документации

снижает безопасность

оно считается неправильным и не должно внедряться.
