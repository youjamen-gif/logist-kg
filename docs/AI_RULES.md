# AI_RULES.md — Logist.kg

Документ определяет обязательные правила для AI-ассистентов и разработчиков, которые создают и изменяют код проекта Logist.kg.

---

# 1. Главная цель

AI работает не как генератор случайного кода, а как строгий архитектурный исполнитель.

Любое изменение в проекте должно быть:

- безопасным
- масштабируемым
- предсказуемым
- согласованным с документацией
- пригодным для production
- понятным следующему AI и человеку-разработчику

AI обязан сохранять целостность платформы Logist.kg как независимой системы.

---

# 2. Базовый технологический стандарт

AI должен считать эталонным следующий стек проекта:

## Frontend
- Next.js (App Router)
- TypeScript
- Tailwind CSS

## Backend
- NestJS
- TypeScript
- Prisma
- PostgreSQL
- JWT Access + Refresh Tokens
- Redis
- Socket.IO

## Storage
- S3 / MinIO-совместимое файловое хранилище

## Dev Principles
- модульная архитектура
- строгая типизация
- DTO + validation
- RBAC
- audit logging
- pagination-first
- API-first

---

# 3. Абсолютные запреты

AI не должен:

- добавлять новую технологию без реальной архитектурной причины
- писать код “на время”, “временно”, “потом исправим”
- дублировать бизнес-логику во frontend и backend
- хранить бизнес-логику в React-компонентах
- писать запросы в БД без понимания индексов, фильтров и пагинации
- ломать существующий контракт API без обновления документации
- менять схему базы без обновления схемных документов
- использовать `any`, если можно описать тип
- отключать guards, validation, ownership checks ради “быстрого запуска”
- оставлять магические строки, магические числа и неочевидные role checks
- создавать хаотичную структуру файлов
- писать код, который зависит от ручного редактирования в 10 местах
- внедрять realtime там, где достаточно обычного HTTP
- смешивать старую и новую архитектуру в одном модуле
- добавлять скрытые зависимости между модулями

---

# 4. Главный принцип изменения проекта

Любое изменение AI обязан делать по следующему порядку:

1. понять, в каком модуле находится задача
2. проверить, не нарушает ли изменение текущую архитектуру
3. проверить, не требует ли изменение правки документации
4. внести изменение минимально необходимым способом
5. сохранить совместимость с остальными модулями
6. не дублировать уже существующие решения
7. оставить код в состоянии, пригодном для следующего этапа разработки

Если архитектура требует изменения документации, AI должен сначала обновить документацию, а потом код, либо отдать обе части вместе.

---

# 5. Источник истины

AI обязан считать источником истины только следующие сущности:

- `docs/PROJECT_STRUCTURE.md` — структура проекта
- `docs/BACKEND_ARCHITECTURE.md` — архитектура backend
- `docs/DATABASE_SCHEMA.md` — логическая схема базы
- `docs/DATABASE_SCHEMA_SQL.md` — SQL-структура
- `docs/API_SPEC_BACKEND.md` — backend API
- `docs/AUTH_FLOW.md` — правила авторизации и доступа
- `docs/ADMIN_PANEL_STRUCTURE.md` — структура админ-панели

Если код противоречит документации, AI не должен молча продолжать хаос. Он обязан приводить проект к целевой архитектуре, а не закреплять ошибку.

---

# 6. Правило единой архитектуры

Logist.kg должен развиваться как единая платформа:

- frontend отвечает только за UI, роутинг, формы, клиентское состояние и вызовы API
- backend отвечает за бизнес-логику, авторизацию, доступ, валидацию, работу с БД, файлы, события, уведомления
- база данных отвечает только за хранение структурированных данных
- файловое хранилище отвечает только за файлы
- Redis отвечает только за кеш, очереди, rate limiting, ephemeral state
- Socket.IO отвечает только за realtime-сценарии

AI не должен переносить обязанности одного слоя в другой.

---

# 7. Структурное правило для репозитория

AI обязан писать код только в целевую актуальную структуру проекта.

## Актуальные зоны:
- `frontend/` — клиентская часть
- `backend/` — серверная часть
- `docs/` — архитектурная документация

## Правило:
- новый frontend-код создаётся внутри `frontend/`
- новый backend-код создаётся внутри `backend/`
- документация меняется только внутри `docs/`

AI не должен развивать legacy-структуру вне целевых директорий, если она осталась от старых этапов.

Если в репозитории есть старые или промежуточные директории, AI не должен использовать их как основание для новой архитектуры.

---

# 8. Правила для frontend

AI должен:

- использовать App Router-подход последовательно
- держать страницы тонкими
- выносить UI-части в переиспользуемые компоненты
- выносить API-вызовы в отдельный слой
- выносить типы в отдельные типизированные файлы
- строить интерфейсы на основе backend contract
- использовать server/client components осознанно
- минимизировать client-only код
- использовать loading / empty / error states
- соблюдать единый layout и общие shell-компоненты
- не дублировать role checks по всему UI без общего механизма

AI не должен:

- держать тяжелую бизнес-логику внутри `page.tsx`
- выполнять критическую проверку прав только на frontend
- обращаться к базе напрямую из frontend
- делать frontend источником истины для ролей, статусов и разрешений
- дублировать одну и ту же разметку на множестве admin-страниц
- смешивать mock-логику и production-логику без явной границы

---

# 9. Правила для backend

AI должен:

- строить backend по модульной структуре NestJS
- использовать controller → service → repository/data-access responsibility split
- использовать DTO для входящих данных
- использовать class-validator для валидации
- использовать guards для авторизации и ролей
- использовать Prisma как единственный слой ORM-доступа к PostgreSQL
- держать бизнес-правила в service layer
- выносить общие проверки в shared guards/decorators/helpers
- разделять public, authenticated, role-protected и admin endpoints
- добавлять audit logging для критических действий
- добавлять нормальную обработку ошибок
- проектировать backend так, чтобы frontend не знал внутренние детали БД

AI не должен:

- писать SQL хаотично по всему коду, если задача решается Prisma
- хранить бизнес-логику в controller
- делать сервисы слишком связанными между собой
- позволять модулю напрямую зависеть от внутренних деталей другого модуля
- смешивать auth, users, admin и domain logic в одном месте
- возвращать наружу внутренние поля, которые не должны видеть клиенты

---

# 10. Правила для базы данных

AI должен исходить из PostgreSQL как основной базы данных.

## Обязательные принципы:
- нормализованная схема
- явные связи
- понятные foreign keys
- корректные enum-значения или их осознанная альтернатива
- индексы для реальных сценариев поиска
- audit-friendly поля
- `createdAt` и `updatedAt` для сущностей, где это нужно
- мягкое удаление только там, где это действительно оправдано
- уникальные ограничения там, где бизнес-логика требует уникальности

AI не должен:

- плодить nullable-поля без причины
- дублировать одно и то же состояние в нескольких таблицах
- хранить массивы/JSON там, где нужна нормальная связь
- делать таблицы “на всякий случай”
- добавлять поля, назначение которых не описано в бизнес-логике
- использовать БД как файловое хранилище

---

# 11. Правила изменения схемы БД

Если AI меняет структуру данных, он обязан:

1. обновить `DATABASE_SCHEMA.md`
2. обновить `DATABASE_SCHEMA_SQL.md`, если меняется SQL-представление
3. обновить `API_SPEC_BACKEND.md`, если меняются DTO/контракты
4. проверить, не ломаются ли frontend-формы, фильтры, детали страниц и admin-панель
5. учитывать миграцию существующих данных

AI не должен менять схему молча.

Любое изменение БД должно быть объяснимо через бизнес-функцию платформы.

---

# 12. Правила API

AI обязан проектировать API как стабильный контракт.

## API должно быть:
- предсказуемым
- REST-последовательным
- версионируемым при необходимости
- типизированным
- защищённым
- документированным

## Обязательные правила:
- списки должны поддерживать pagination
- фильтры должны быть явными
- сортировка должна быть явной
- ответы должны быть консистентными
- ошибки должны быть нормализованы
- приватные поля не должны уходить в публичные ответы
- admin endpoints должны быть отделены от user endpoints
- ownership и role checks должны применяться на backend

AI не должен:
- делать “специальные” endpoints без архитектурной причины
- создавать дублирующие endpoints с почти одинаковым смыслом
- возвращать сырые ORM-объекты без контроля выходной модели
- ломать naming consistency

---

# 13. Правила авторизации и доступа

AI обязан соблюдать RBAC и ownership model.

## Роли платформы:
- guest
- driver
- shipper
- dispatcher
- admin

Если в конкретном этапе какая-то роль ещё не активирована кодом, AI всё равно обязан учитывать целевую модель ролей в архитектуре.

## Обязательные правила:
- guest не получает доступ к защищённым действиям
- driver не получает admin-права
- shipper не должен управлять чужими грузами
- driver не должен редактировать чужие отклики
- admin-права не должны определяться на frontend
- все чувствительные операции защищаются на backend
- доступ к документам, файлам и модерации должен проверяться сервером

AI не должен:
- полагаться только на скрытие кнопки в UI
- считать пользователя владельцем сущности без проверки на backend
- смешивать “роль” и “статус верификации”
- давать универсальный доступ “для теста”

---

# 14. Правила статусов и бизнес-состояний

AI должен различать:

- роль пользователя
- статус аккаунта
- статус верификации
- статус груза
- статус отклика
- статус оплаты
- статус модерации
- статус публикации

Эти состояния нельзя смешивать в одно поле.

Каждое состояние должно иметь:
- явный список значений
- понятные переходы
- одинаковую трактовку во frontend, backend и документации

---

# 15. Правила файлов и документов

AI должен считать, что файлы хранятся вне базы данных.

Разрешено хранить в БД только метаданные:

- `url`
- `path`
- `mimeType`
- `size`
- `originalName`
- `uploadedBy`
- `entityType`
- `entityId`

AI должен:
- разделять публичные и приватные файлы
- учитывать права доступа к документам водителей и компаний
- валидировать типы и размеры файлов
- предусматривать безопасные имена и пути
- хранить только необходимые ссылки и метаданные

AI не должен:
- хранить бинарные файлы в PostgreSQL без критической причины
- открывать прямой доступ ко всем документам
- смешивать документы верификации и публичные медиа в одном неуправляемом пространстве

---

# 16. Правила realtime

Realtime использовать только там, где он реально нужен.

Разрешённые realtime-сценарии:
- чат
- live-уведомления
- аукционные/торговые обновления
- служебные admin/live события при необходимости

AI должен:
- использовать Socket.IO точечно
- проектировать события с понятными именами
- проверять авторизацию сокетов
- ограничивать подписки по пользователю, диалогу, комнате или сущности
- предусматривать fallback через обычный HTTP

AI не должен:
- переводить весь проект на realtime
- открывать глобальные каналы без ограничений
- слать лишние данные в событиях
- дублировать одно и то же событие множеством способов

---

# 17. Правила производительности

AI обязан по умолчанию проектировать код с расчётом на рост проекта.

## Обязательные принципы:
- pagination-first
- filter-first
- index-aware queries
- lean response payloads
- отсутствие лишних joins/include
- отсутствие лишних API-вызовов
- отсутствие дублирования данных в ответах
- кеширование только там, где оно оправдано
- тяжёлые задачи — в очереди или фоне, а не в request lifecycle

AI не должен:
- тянуть весь список без лимита
- строить интерфейс на полном наборе данных
- делать N+1 запросы без необходимости
- грузить неподготовленные admin-таблицы без пагинации и фильтров
- добавлять тяжёлые вычисления в рендер страницы

---

# 18. Правила admin-панели

Admin-панель — это отдельный управленческий слой, а не набор случайных страниц.

AI должен:
- сохранять единую структуру admin routes
- использовать общий admin layout
- использовать единые таблицы, фильтры, actions, headers
- отделять просмотр, модерацию, редактирование и destructive actions
- добавлять подтверждение для критических действий
- логировать admin-действия
- делать bulk actions только там, где они безопасны
- проектировать admin UI на основе backend permissions

AI не должен:
- копировать одинаковый код между admin-страницами
- смешивать public UI и admin UI
- выполнять критические admin-действия без audit trail
- скрывать сложность системы за неявным поведением

---

# 19. Правила логирования и аудита

Все критические действия должны быть отслеживаемыми.

AI обязан логировать:
- создание и удаление грузов
- изменение статусов
- блокировку и разблокировку пользователей
- модерацию документов
- изменения системных настроек
- финансовые действия
- admin-действия
- спорные и чувствительные операции

Каждая запись аудита должна по возможности содержать:
- кто выполнил действие
- над какой сущностью
- какой был тип действия
- когда произошло действие
- ключевые изменения
- IP / user agent при необходимости и допустимости
- ссылку на связанный объект

AI не должен:
- писать audit logging “потом”
- логировать слишком мало для расследования проблем
- хранить секреты, токены и пароли в логах

---

# 20. Правила безопасности

AI обязан проектировать безопасно по умолчанию.

## Обязательные меры:
- JWT access token + refresh token strategy
- refresh token rotation
- password hashing
- DTO validation
- role guards
- ownership checks
- rate limiting
- безопасная обработка файлов
- защита admin endpoints
- защита websocket connections
- минимально необходимые данные в ответах
- secrets только через env/config

AI не должен:
- хранить пароли в открытом виде
- возвращать чувствительные поля в API
- доверять данным клиента без серверной проверки
- писать `TODO: validate later` в критических местах
- оставлять тестовые ключи и секреты в коде
- использовать `.env.local` как источник production-значений в репозитории

---

# 21. Правила качества кода

AI должен генерировать код уровня senior production baseline.

## Обязательные требования:
- осмысленные имена файлов, классов, функций и переменных
- единый стиль именования
- небольшие понятные функции
- отсутствие дублирования
- выделение shared utilities только при реальной повторяемости
- комментарии только там, где без них решение неочевидно
- строгая типизация
- понятные интерфейсы между слоями
- минимально необходимая сложность

AI не должен:
- засорять проект бесполезными helper-файлами
- писать нечитаемые универсальные абстракции
- создавать “монстр-компоненты” и “монстр-сервисы”
- размазывать один use case по множеству случайных мест

---

# 22. Правила миграции legacy-кода

Если AI сталкивается со старым кодом, он должен:

- сохранять работоспособность текущего этапа
- не распространять legacy-паттерны на новые модули
- постепенно переносить проект на целевую архитектуру
- не смешивать старый Firebase-подход и новый backend-подход в одной новой функции
- предпочитать адаптацию через API-слой, а не хаотичное прямое переписывание всего сразу

AI не должен:
- укреплять временные решения как постоянные
- строить новый функционал поверх устаревшей архитектуры
- оставлять неоднозначность, какой слой теперь основной

---

# 23. Правила перед выдачей ответа или кода

Перед тем как предложить код, AI обязан проверить:

1. в правильную ли директорию он пишет
2. не противоречит ли решение архитектуре
3. не создаёт ли решение новую скрытую зависимость
4. не требует ли задача обновления документации
5. не дублирует ли решение уже существующий механизм
6. не переносит ли backend-логику во frontend
7. не нарушает ли права доступа
8. не создаёт ли решение проблем с масштабированием
9. не ухудшает ли читаемость проекта
10. можно ли это решение безопасно развивать дальше

---

# 24. Когда AI обязан обновлять документацию

AI обязан обновлять документы, если меняется хотя бы одно из следующих:

- структура каталогов
- состав модулей
- схема БД
- роли и доступы
- API endpoints
- жизненный цикл авторизации
- структура admin-панели
- правила загрузки и хранения файлов
- ключевые бизнес-состояния
- стратегия realtime
- стратегия payments/subscriptions
- ключевые ограничения системы

---

# 25. Финальный принцип

AI не должен быть “быстрым генератором кода”.
AI должен быть системным архитектором-исполнителем.

Любое решение для Logist.kg должно отвечать сразу на 5 вопросов:

- это безопасно?
- это масштабируемо?
- это соответствует целевой архитектуре?
- это не усложняет проект без причины?
- следующий AI или разработчик сможет это поддерживать?

Если ответ хотя бы на один вопрос “нет”, решение считается плохим и не должно предлагаться как финальное.
