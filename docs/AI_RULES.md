# AI_RULES.md — Logist.kg

Документ описывает правила для AI-ассистентов и разработчиков при работе с кодом проекта Logist.kg.

Цель:
- сохранить архитектуру проекта
- избежать ошибок в базе
- не увеличивать расходы Firebase
- не ломать структуру платформы

---

# 1. Общие правила разработки

AI не должен:

- менять структуру базы без обновления DATABASE_SCHEMA.md
- добавлять новые коллекции без необходимости
- использовать неиндексированные поля для поиска
- писать небезопасные Firestore rules
- дублировать данные в разных коллекциях
- создавать массивы больших размеров в документах

AI должен:

- использовать существующую архитектуру
- следовать PROJECT_STRUCTURE.md
- соблюдать FIRESTORE_QUERIES.md
- соблюдать FIREBASE_COST_RULES.md
- соблюдать FIRESTORE_RULES.md

---

# 2. Работа с Firestore

AI должен:

- использовать `limit()` в списках
- использовать `orderBy()` для сортировки
- использовать `where()` для фильтрации
- использовать пагинацию

AI не должен:

- загружать всю коллекцию
- использовать realtime для больших списков
- хранить файлы в Firestore
- хранить большие тексты в индексах

---

# 3. Работа с коллекциями

AI должен использовать только существующие коллекции:

---

# 4. Работа с ролями

Поддерживаемые роли:


driver
shipper
dispatcher
admin


AI должен:

- проверять роль пользователя
- ограничивать действия по ролям
- не давать доступ к админ функциям обычным пользователям

---

# 5. Безопасность

AI обязан:

- проверять `request.auth`
- проверять владельца документа
- соблюдать Firestore rules

AI не должен:

- писать `allow read, write: if true`
- открывать базу полностью
- давать доступ к чужим документам

---

# 6. Работа с Storage

AI должен:

- хранить файлы только в Firebase Storage
- сохранять только URL в Firestore

Папки Storage:


avatars/
driver-documents/
company-documents/
vehicle-documents/
chat-files/
banners/
ads/


---

# 7. Производительность

AI должен:

- использовать индексы
- использовать pagination
- минимизировать realtime подписки

AI не должен:

- делать сложные запросы без индексов
- делать частые обновления документов
- использовать тяжелые запросы

---

- совместимым с существующей архитектурой

# 13. Firebase Cost Pitfalls (Критические ошибки)

AI и разработчики должны избегать следующих ошибок при работе с Firebase.

---

## 13.1 Realtime на большие коллекции

Нельзя подписываться на всю коллекцию.

❌ Неправильно

```js
onSnapshot(collection(db, "freights"))

Это вызывает тысячи чтений при каждом изменении.

✅ Правильно

подписываться только на конкретный документ

использовать limit()

использовать pagination

13.2 Загрузка всей коллекции

❌ Неправильно

db.collection("freights").get()

Если в базе 50 000 грузов — это 50 000 read операций.

✅ Правильно

db.collection("freights")
.limit(20)

Использовать:

pagination

startAfter()

13.3 Большие массивы в документах

❌ Неправильно

{
  "bids": [ ...1000 элементов... ]
}

Каждое чтение документа загружает весь массив.

✅ Правильно

Создавать отдельную коллекцию:

freights
bids
13.4 Хранение файлов в Firestore

❌ Нельзя хранить:

base64

изображения

pdf

Firestore предназначен только для данных.

✅ Использовать Firebase Storage

Firestore хранит только:

fileUrl
filePath
fileType
13.5 Слишком много индексов

Firebase часто предлагает создать индекс автоматически.

AI не должен создавать индексы без необходимости.

Разрешённые индексы для проекта:

freights
originCity + destinationCity

freights
status + createdAt

bids
freightId + createdAt
13.6 Частое обновление документов

❌ Нельзя обновлять документ слишком часто.

Например:

счётчики в реальном времени

обновление статуса каждую секунду

Это увеличивает стоимость writes.

13.7 Неправильные realtime подписки

Realtime разрешён только для:

чатов

уведомлений

конкретного груза

Realtime запрещён для:

списков грузов

списка пользователей

больших таблиц
