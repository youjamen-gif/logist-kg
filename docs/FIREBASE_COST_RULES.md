# FIREBASE_COST_RULES.md — Logist.kg

## 1. Главная цель

Минимизировать расходы в Firebase без потери скорости, безопасности и возможности масштабирования.

---

## 2. Основные правила экономии

### 2.1. Не читать лишние документы
- всегда использовать `where()`
- всегда использовать `limit()`
- не загружать всю коллекцию целиком
- не делать `get()` без фильтров на больших коллекциях

### 2.2. Осторожно с realtime
- не использовать realtime listeners на всю коллекцию `freights`
- не подписываться на тысячи документов одновременно
- realtime использовать только там, где это действительно нужно:
  - чат
  - уведомления
  - точечные статусы

### 2.3. Не хранить лишние поля
- не дублировать большие объекты
- не хранить повторяющиеся данные в нескольких коллекциях
- хранить только нужные поля для списка
- подробные данные загружать отдельно

### 2.4. Не хранить файлы в Firestore
- Firestore только для метаданных
- файлы хранить только в Storage
- в Firestore сохранять только URL, path, fileType, size

### 2.5. Не делать слишком много индексов
- индексировать только те поля, которые реально участвуют в поиске и сортировке
- не индексировать длинные тексты
- не создавать лишние composite indexes

---

## 3. Правильные подходы к коллекциям

### users
- хранить только основную информацию
- документы верификации не хранить как большие объекты
- ссылки на файлы хранить отдельно

### freights
- список грузов должен быть лёгким
- не хранить большие вложенные массивы
- не хранить список всех откликов внутри груза
- `bidsCount` хранить числом, сами отклики отдельно в `bids`

### bids
- каждый отклик отдельным документом
- не хранить отклики внутри `freights`

### messages
- сообщения отдельной коллекцией
- не хранить весь чат одним документом

---

## 4. Чего нельзя делать

- нельзя делать `onSnapshot()` на все грузы
- нельзя хранить массивы на сотни и тысячи элементов в одном документе
- нельзя делать сложный поиск без индексов
- нельзя загружать все поля, если нужен только список
- нельзя хранить base64 файлов в документах
- нельзя обновлять один и тот же документ слишком часто без необходимости

---

## 5. Правильные паттерны

### Пагинация
Использовать:
- `limit()`
- `startAfter()`

### Денормализация
Разрешена только контролируемая:
- `bidsCount`
- `views`
- `rating`
- `reviewsCount`

### Разделение списка и деталей
Для списка грузов использовать только:
- маршрут
- вес
- цена
- дата
- тип кузова
- статус

Полное описание грузить только на странице детали.

---

## 6. Самые дорогие места в Firebase

### Firestore reads
Особенно дорого, если:
- часто обновлять списки
- делать лишние запросы
- использовать realtime без ограничений

### Storage
Дорого, если:
- хранить большие неиспользуемые файлы
- не удалять старые документы
- загружать оригиналы без сжатия

### Functions
Дорого, если:
- вызывать слишком часто
- делать тяжёлую логику на каждый запрос
- запускать бесконтрольные триггеры

---

## 7. Что использовать осторожно

- Cloud Functions triggers
- массовые обновления документов
- realtime dashboards
- full-text search внутри Firestore
- сложные фильтры по многим полям одновременно

---

## 8. Практические правила для Logist.kg

### Для поиска грузов
- сначала фильтры по маршруту
- потом дата
- потом тип кузова
- потом лимит 20–50 документов

### Для чата
- realtime только для активного диалога
- старые сообщения грузить пагинацией

### Для уведомлений
- хранить короткие записи
- старые уведомления архивировать или удалять

### Для документов
- хранить в Storage по папкам:
  - `driver-documents/`
  - `company-documents/`
  - `vehicle-documents/`

---

## 9. Рекомендованные лимиты

- грузы в одном запросе: 20–50
- сообщения в одном запросе: 30–50
- уведомления в одном запросе: 20–30
- отзывы в одном запросе: 10–20

---

## 10. Контроль расходов

Регулярно проверять:
- Firestore Reads
- Firestore Writes
- Storage usage
- Functions invocations
- самые дорогие запросы
- ненужные индексы
- старые файлы в Storage

---

## 11. Правило проекта

Каждая новая функция должна проверяться по 3 вопросам:

1. сколько будет чтений?
2. можно ли уменьшить количество realtime подписок?
3. можно ли хранить это дешевле и проще?

Если ответ плохой — архитектуру менять до внедрения.
