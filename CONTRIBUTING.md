# CONTRIBUTING.md — Logist.kg

Этот документ описывает правила для разработчиков и AI-ассистентов, которые работают с проектом Logist.kg.

Цель:
- сохранить архитектуру проекта
- избежать ошибок в базе данных
- не увеличивать расходы Firebase
- поддерживать единый стиль разработки

---

# 1. Основные принципы

Перед добавлением новой функции всегда проверяйте:

1. соответствует ли изменение архитектуре проекта
2. не увеличивает ли это стоимость Firebase
3. не нарушает ли структуру базы
4. не ломает ли существующие API

Если ответ хотя бы на один вопрос отрицательный — архитектуру нужно пересмотреть.

---

# 2. Работа с GitHub

## Ветки

Используются две основные ветки:

main  
Стабильная версия проекта.

dev  
Основная ветка разработки.

Все изменения сначала добавляются в **dev**, затем после проверки попадают в **main**.

---

## Pull Request

Перед объединением изменений необходимо:

- проверить код
- проверить Firestore запросы
- проверить безопасность
- убедиться, что не добавлены лишние зависимости

---

# 3. Работа с Firestore

## Запросы

Нужно всегда использовать:

- where()
- limit()
- orderBy()
- pagination

Нельзя:

- загружать всю коллекцию
- использовать realtime для больших списков
- делать сложные запросы без индексов

---

## Структура базы

Использовать только существующие коллекции:

drivers  
companies

freights

conversations

notifications

Новые коллекции можно добавлять только после обновления:

---

# 4. Работа с Firebase Storage

Файлы должны храниться только в Storage.

Firestore может хранить только:

- fileUrl
- filePath
- fileType

Папки Storage:

avatars/  
driver-documents/  
company-documents/  
vehicle-documents/  
chat-files/  
banners/  
ads/

---

# 5. Безопасность

Каждый запрос должен учитывать:

- роль пользователя
- владельца документа
- Firestore security rules

Нельзя:

- писать allow read, write: if true
- открывать доступ к чужим данным
- давать доступ к admin функциям обычным пользователям

---

# 6. Работа с ролями

Поддерживаемые роли:

driver

dispatcher  
admin  

Каждая роль имеет свои ограничения и доступы.

# 9. Логирование

Все важные действия должны записываться в:

audit_logs

Например:

- создание груза
- удаление груза
- блокировка пользователя
- изменение системных настроек

---

# 10. Главный принцип проекта

Каждая новая функция должна быть:

- безопасной
- масштабируемой
- дешёвой для Firebase
- понятной для будущих разработчиков
